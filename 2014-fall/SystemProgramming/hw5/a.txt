			<h1 id="firstHeading" class="firstHeading" lang="en"><span dir="auto">Readers–writers problem</span></h1>

<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, the <b>first and second readers-writers problems</b> are examples of a common computing problem in <a href="/wiki/Concurrency_(computer_science)" title="Concurrency (computer science)">concurrency</a>. The two problems deal with situations in which many <a href="/wiki/Thread_(computer_science)" title="Thread (computer science)" class="mw-redirect">threads</a> must access the same <a href="/wiki/Shared_memory" title="Shared memory">shared memory</a> at one time, some reading and some writing, with the natural constraint that no process may access the share for reading or writing while another process is in the act of writing to it. (In particular, it <i>is</i> allowed for two or more readers to access the share at the same time.) A <a href="/wiki/Readers-writer_lock" title="Readers-writer lock" class="mw-redirect">readers-writer lock</a> is a <a href="/wiki/Data_structure" title="Data structure">data structure</a> that solves one or more of the readers-writers problems.</p>

<h2><span class="mw-headline" id="The_first_readers-writers_problem">The first readers-writers problem</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Readers%E2%80%93writers_problem&amp;action=edit&amp;section=1" title="Edit section: The first readers-writers problem">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Suppose we have a shared memory area with the constraints detailed above. It is possible to protect the shared data behind a mutual exclusion <a href="/wiki/Mutex" title="Mutex" class="mw-redirect">mutex</a>, in which case no two threads can access the data at the same time. However, this solution is suboptimal, because it is possible that a reader <i>R<sub>1</sub></i> might have the lock, and then another reader <i>R<sub>2</sub></i> requests access. It would be foolish for <i>R<sub>2</sub></i> to wait until <i>R<sub>1</sub></i> was done before starting its own read operation; instead, <i>R<sub>2</sub></i> should start right away. This is the motivation for the <b>first readers-writers problem</b>, in which the constraint is added that <i>no reader shall be kept waiting if the share is currently opened for reading.</i> This is also called <b>readers-preference</b>, with its solution below:</p>
<div dir="ltr" class="mw-geshi mw-code mw-content-ltr">
<div class="c source-c">
<pre class="de1">
semaphore wrt<span class="sy0">=</span><span class="nu0">1</span><span class="sy0">,</span> mutex<span class="sy0">=</span><span class="nu0">1</span><span class="sy0">;</span>
readcount<span class="sy0">=</span><span class="nu0">0</span><span class="sy0">;</span>
 
writer<span class="br0">(</span><span class="br0">)</span>
<span class="br0">{</span>
    wait<span class="br0">(</span>wrt<span class="br0">)</span><span class="sy0">;</span>
    <span class="co1">// Writing is done</span>
    signal<span class="br0">(</span>wrt<span class="br0">)</span><span class="sy0">;</span>
<span class="br0">}</span>
 
reader<span class="br0">(</span><span class="br0">)</span>
<span class="br0">{</span>
    wait<span class="br0">(</span>mutex<span class="br0">)</span><span class="sy0">;</span>
        readcount<span class="sy0">++;</span>
        <span class="kw1">if</span> <span class="br0">(</span>readcount <span class="sy0">==</span> <span class="nu0">1</span><span class="br0">)</span>
            wait<span class="br0">(</span>wrt<span class="br0">)</span><span class="sy0">;</span>
    signal<span class="br0">(</span>mutex<span class="br0">)</span><span class="sy0">;</span>
    <span class="co1">// Do the Reading</span>
    <span class="co1">// (Critical Section Area)</span>
    wait<span class="br0">(</span>mutex<span class="br0">)</span><span class="sy0">;</span>
        readcount<span class="sy0">--;</span>
        <span class="kw1">if</span> <span class="br0">(</span>readcount <span class="sy0">==</span> <span class="nu0">0</span><span class="br0">)</span>
            signal<span class="br0">(</span>wrt<span class="br0">)</span><span class="sy0">;</span>
    signal<span class="br0">(</span>mutex<span class="br0">)</span><span class="sy0">;</span>
<span class="br0">}</span>
</pre></div>
</div>
<h2><span class="mw-headline" id="The_second_readers-writers_problem">The second readers-writers problem</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Readers%E2%80%93writers_problem&amp;action=edit&amp;section=2" title="Edit section: The second readers-writers problem">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Above solution is suboptimal, because it is possible that a reader <i>R<sub>1</sub></i> might have the lock, a writer <i>W</i> be waiting for the lock, and then a reader <i>R<sub>2</sub></i> requests access. It would be foolish for <i>R<sub>2</sub></i> to jump in immediately, ahead of <i>W</i>; if that happened often enough, <i>W</i> would <a href="/wiki/Resource_starvation" title="Resource starvation">starve</a>. Instead, <i>W</i> should start as soon as possible. This is the motivation for the <b>second readers-writers problem</b>, in which the constraint is added that <i>no writer, once added to the queue, shall be kept waiting longer than absolutely necessary</i>. This is also called <b>writers-preference</b>.</p>
<p>A solution to the <b>writers-preference</b> scenario is presented below:.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1"><span>[</span>1<span>]</span></a></sup> In this code, the semaphore mutex_rdcnt protects the readcount variable, mutex_wrcnt protects the writecount variable, the mutex 'r' protects the reading operation, and the mutex 'w' protects the writing operation.</p>
<div dir="ltr" class="mw-geshi mw-code mw-content-ltr">
<div class="ada source-ada">
<pre class="de1">
int readcount, writecount; //<span class="br0">(</span>initial value = <span class="nu0">0</span><span class="br0">)</span>
semaphore mutex_rdcnt, mutex_wrcnt, mutex_3, w, r; //<span class="br0">(</span>initial value = <span class="nu0">1</span><span class="br0">)</span>
 
//READER
  wait<span class="br0">(</span>mutex_3<span class="br0">)</span>;
    wait<span class="br0">(</span>r<span class="br0">)</span>;
      wait<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;
        readcount++;
        <span class="kw1">if</span> <span class="br0">(</span>readcount == <span class="nu0">1</span><span class="br0">)</span>
            wait<span class="br0">(</span>w<span class="br0">)</span>;
      signal<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;
    signal<span class="br0">(</span>r<span class="br0">)</span>;
  signal<span class="br0">(</span>mutex_3<span class="br0">)</span>;
 
 // reading <span class="kw1">is</span> performed
 
  wait<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;
    readcount<span class="co1">--;</span>
    <span class="kw1">if</span> <span class="br0">(</span>readcount == <span class="nu0">0</span><span class="br0">)</span> 
        signal<span class="br0">(</span>w<span class="br0">)</span>;
  signal<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;
 
//WRITER
  wait<span class="br0">(</span>mutex_wrcnt<span class="br0">)</span>;
    writecount++;
    <span class="kw1">if</span> <span class="br0">(</span>writecount == <span class="nu0">1</span><span class="br0">)</span> 
        wait<span class="br0">(</span>r<span class="br0">)</span>;
  signal<span class="br0">(</span>mutex_wrcnt<span class="br0">)</span>;
 
  wait<span class="br0">(</span>w<span class="br0">)</span>;
   // writing <span class="kw1">is</span> performed
  signal<span class="br0">(</span>w<span class="br0">)</span>;
 
  wait<span class="br0">(</span>mutex_wrcnt<span class="br0">)</span>;
    writecount<span class="co1">--;</span>
    <span class="kw1">if</span> <span class="br0">(</span>writecount == <span class="nu0">0</span><span class="br0">)</span> 
        signal<span class="br0">(</span>r<span class="br0">)</span>;
  signal<span class="br0">(</span>mutex_wrcnt<span class="br0">)</span>;
</pre></div>
</div>
<h2><span class="mw-headline" id="The_third_readers-writers_problem">The third readers-writers problem</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Readers%E2%80%93writers_problem&amp;action=edit&amp;section=3" title="Edit section: The third readers-writers problem">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In fact, the solutions implied by both problem statements result in starvation — the first readers-writers problem may starve writers in the queue, and the second readers-writers problem may starve readers. Therefore, the <b>third readers-writers problem</b> is sometimes proposed, which adds the constraint that <i>no thread shall be allowed to starve</i>; that is, the operation of obtaining a lock on the shared data will always terminate in a bounded amount of time. A solution with fairness for both readers and writers might be as follows:</p>
<div dir="ltr" class="mw-geshi mw-code mw-content-ltr">
<div class="ada source-ada">
<pre class="de1">
int readcount; // <span class="br0">(</span>initial value = <span class="nu0">0</span><span class="br0">)</span>
semaphore mutex_rdcnt, r, w; // <span class="br0">(</span> initial value = <span class="nu0">1</span> <span class="br0">)</span> 
 
//READER
        wait<span class="br0">(</span>r<span class="br0">)</span>;
          wait<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;
            readcount++;
            <span class="kw1">if</span> <span class="br0">(</span>readcount == <span class="nu0">1</span><span class="br0">)</span>
              wait<span class="br0">(</span>w<span class="br0">)</span>;
          signal<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;          
        signal<span class="br0">(</span>r<span class="br0">)</span>;
 
        // reading <span class="kw1">is</span> performed 
 
        wait<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;
          readcount<span class="co1">--;</span>
          <span class="kw1">if</span> <span class="br0">(</span>readcount == <span class="nu0">0</span><span class="br0">)</span>
            signal<span class="br0">(</span>w<span class="br0">)</span>;
        signal<span class="br0">(</span>mutex_rdcnt<span class="br0">)</span>;  
 
//WRITER
        wait<span class="br0">(</span>r<span class="br0">)</span>;
        wait<span class="br0">(</span>w<span class="br0">)</span>;
        signal<span class="br0">(</span>r<span class="br0">)</span>;
 
         // writing <span class="kw1">is</span> performed
 
        signal<span class="br0">(</span>w<span class="br0">)</span>;
</pre></div>
</div>
<p>Note that sections protected by mutex_rdcnt could be replaced by a suitable <a href="/wiki/Fetch-and-add" title="Fetch-and-add">fetch-and-add</a> atomic instruction, saving two potential context switches in reader's code.</p>
<p>Note also that this solution can only satisfy the condition that "no thread shall be allowed to starve" if and only if semaphores preserve first-in first-out ordering when blocking and releasing threads. Otherwise, a blocked writer, for example, may remain blocked indefinitely with a cycle of other writers decrementing the semaphore before it can.</p>
